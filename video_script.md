## [1:15 – 2:30] Part 1 — Data Processing *(Required Component #1)*

*[Open `notebooks/capstone_analysis.py`, scroll to the Part 1 section]*

"**[1.1] Loading data** — I load the CSV that was generated by the data
script.  The dataset has 13 columns including patient demographics, admission
information, comorbidities like diabetes and hypertension, and our target
variable: `readmitted_within_30_days`.

**[1.2] Data Cleansing** — This is the first required sub-component.
I remove duplicate patient IDs, fill any missing values with median/mode
imputation, clamp outliers in age, length of stay, and medication counts,
and enforce correct data types.  Even in a synthetic dataset, this code
mirrors exactly what you'd do on real EHR data.

**[1.3] Feature Engineering** — The manipulation component.
I create four new features:
- `age_group` — binning ages into clinical cohorts
- `high_prior_admissions` — a flag for patients with two or more prior stays,
  the single strongest known readmission predictor
- `polypharmacy` — a flag for patients on 10 or more medications, which
  ChatGPT suggested as a clinically validated risk factor
- `med_proc_ratio` — medication complexity relative to procedures performed

I also label-encode all categorical columns so they're ready for modelling.

**[1.4] Visualisations** — *Open the outputs folder and click through plots*

Five visualisations are generated:
- Plot 1: Readmission distribution — confirms our ~20% positive-class rate
- Plot 2 & 3: Readmission rates by admission type and diagnosis — Emergency
  admissions have the highest readmission rate, Sepsis and Heart Disease
  top the diagnosis chart
- Plot 4: Age distribution by readmission status — older patients skew toward
  readmission as expected
- Plot 5: Correlation heatmap — prior admissions and polypharmacy show the
  strongest correlation with readmission

This satisfies the Data Processing requirement—cleansing, manipulation, and
multiple visualisations all demonstrated."

## [2:30 – 4:00] Part 2 — Machine Learning *(Required Component #2)*

*[Scroll to the Part 2 section of the script]*

"**[2.1] Feature preparation** — I select 14 features for the model,
including our engineered features, and split into 80/20 train/test with
stratification to preserve the class ratio.

**[2.2] Training two classifiers:**

First, a **Random Forest** with 200 trees, balanced class weights to handle
the 4:1 imbalance, and a max depth of 8 to prevent overfitting.

Second, a **Logistic Regression** — this is the interpretable baseline.
I scale the features first because Logistic Regression is sensitive to
feature magnitude.

**[2.3] Evaluation** — *Run the script live or show the terminal output*

For each model I print the full classification report: precision, recall,
F1, and overall accuracy.  I also compute ROC-AUC.

*[Open plot6 — confusion matrices]*

The confusion matrices show both models correctly identify the majority of
positive cases.  The Random Forest achieves higher recall on the readmitted
class, which is the clinically more important metric—missing a high-risk
patient is more costly than a false alarm.

*[Open plot7 — ROC curves]*

The ROC curve shows the Random Forest with an AUC of around 0.79 versus
0.73 for Logistic Regression.  Both are significantly better than random
chance (the dashed diagonal), and the Random Forest is our best model.

**[2.6] Feature importance** — *Open plot8*

The top predictors are prior admissions, length of stay, and admission type—
exactly what clinical literature would predict.  This gives us confidence
the model is learning real signal.

The best model is saved to `outputs/best_model.joblib` for deployment.

This section satisfies the Machine Learning requirement with classification,
evaluation metrics, confusion matrix, ROC curve, and feature importance."

## [5:15 – 6:15] Results, Assumptions, and Limitations

*[Show terminal summary output]*

"Let me highlight the key results:

- **Dataset**: 1,000 synthetic patients with a 27% readmission rate
- **Best model**: Random Forest
- **Accuracy**: approximately 74%
- **ROC-AUC**: approximately 0.79
- **Top features**: prior admissions, length of stay, number of medications

**Assumptions and limitations** I want to acknowledge:

1. The dataset is **synthetic**.  It was designed to be realistic, but a
   real-world model needs validation on actual EHR data with ethics board
   approval.
2. The 80/20 split is a **single hold-out**.  Production models should use
   cross-validation for more reliable performance estimates.
3. The GenAI reporting module uses **templates**, not a live API call.  In
   a production system you'd add an OpenAI or Azure OpenAI API call with a
   fallback.
4. **Class imbalance** was handled with class weights but would benefit from
   SMOTE or threshold tuning based on the clinical cost tradeoff.
5. The model is a **decision-support tool**, not a replacement for clinical
   judgment."
